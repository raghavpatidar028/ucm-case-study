Section 03 — Technical Plan & Dashboard Design Programmer/Analyst · UAIR
Section 03

My Technical Approach — Four Pillars

Each pillar addresses a specific challenge identified in Section 02. The plan is designed to be implementable incrementally — delivering value at each stage without requiring a full system overhaul before anything ships.

01
ETL & Database Optimization (Banner + Workday)
  • Design Python + SQL Stored Procedure ETL jobs extracting student records from Banner and HR/finance data from Workday into a unified staging schema
  • Implement incremental load processes to minimize load on production systems during off-peak hours
  • Build automated data validation scripts checking record counts, null rates, and cross-system key integrity before each load
  • Design a metadata catalog and data dictionary aligned with IPEDS field definitions for compliance reporting
  • Establish quarterly data audit routines with exception reports delivered to UAIR leadership
02
Power BI Dashboard Suite
  • Student Success Dashboard for academic advisors — at-risk score, term GPA trend, course engagement, early-alert flags by advisee
  • Retention & Enrollment Trends Dashboard for leadership — term-to-term retention, cohort graduation tracking, enrollment by college and demographic
  • Compliance Reporting Dashboard for UAIR staff — IPEDS submission tracker, data quality scorecard, field completeness rates
  • Implement Row-Level Security (RLS) so advisors see only their own advisees; deans see only their college
  • Standardized DAX measure library: Term-to-Term Retention Rate, Course Drop Ratio, Major Switching Trend, At-Risk Score
03
Predictive Retention Modeling
  • Build logistic regression baseline model using term GPA, credit hours, major, engagement signals, and prior retention history
  • Develop gradient boosting (XGBoost) model to improve at-risk prediction precision — targeting 80%+ early identification accuracy
  • Connect model outputs directly to Power BI via Python scripting connector for live advisor dashboards
  • Design a tiered alert framework: Green (on-track), Yellow (monitor), Red (immediate outreach) per student per term
  • Validate model bias across demographic groups — ensure predictions don't amplify existing inequities in support resources
04
Documentation, Governance & Enablement
  • Document every ETL pipeline with lineage maps showing Banner field → staging table → Power BI measure
  • Write a UCM Data Dictionary aligned with IPEDS definitions, stored in Confluence or SharePoint for campus-wide access
  • Develop self-service training guides for faculty and advisors — how to use dashboards, how to interpret at-risk scores
  • Establish a data request triage process with SLA commitments to reduce UAIR's ad-hoc burden by 30%+
  • Create onboarding documentation for new UAIR analysts covering all pipeline logic, refresh schedules, and governance policies
